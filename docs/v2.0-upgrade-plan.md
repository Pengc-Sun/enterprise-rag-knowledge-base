# v2.0 Workspace Upgrade Plan

This document captures the v2.0 upgrade requirements and turns them into an implementation plan for evolving Enterprise RAG Knowledge Base from a knowledge-base chat app into a workspace-based AI analysis platform.

## v2.0 Acceptance Criteria

- Users can create multiple workspaces.
- A workspace can be created from a template.
- Templates automatically create directory structure, analysis tasks, and report sections.
- Documents, chunks, and conversations are fully isolated by workspace.
- Different templates define different AI analysis tasks.
- AI output is saved as structured results with citations.
- Reviewers can approve, edit, or reject AI conclusions.
- Formal reports can only use reviewed and approved content.
- Reports can be exported as Markdown, DOCX, and PDF.
- Workspace dashboards show project status.
- Existing v1.0 data migrates successfully without data loss.
- Automated tests cover cross-workspace data isolation.
- Docker supports both fresh install and database upgrade paths.
- GitHub ships a v2.0.0 release.

## Current v1.0 Baseline

The current system is knowledge-base centric:

```text
User
-> KnowledgeBase
-> Document
-> DocumentChunk
-> Conversation
-> Message
```

Current isolation is mostly enforced through `knowledge_base_id` and knowledge-base memberships. Retrieval already filters chunks by `DocumentChunk.knowledge_base_id`, which is a useful foundation, but v2.0 needs a stronger top-level tenant boundary:

```text
Workspace
-> WorkspaceMember
-> KnowledgeBase
-> Document
-> DocumentChunk
-> Conversation
-> Message
-> AnalysisTask
-> AnalysisResult
-> ReviewDecision
-> Report
```

## Target Product Shape

v2.0 should support workspace-driven business workflows, not only ad hoc chat.

A workspace represents one business analysis project, for example:

- Company policy review workspace
- IT support knowledge workspace
- Legal contract review workspace
- Sales enablement workspace
- Research literature analysis workspace

A template defines what gets created inside a workspace:

- directories
- default knowledge bases or document categories
- analysis tasks
- report outline
- reviewer workflow rules
- expected AI output schemas

## Core Domain Model

### Workspace

Represents a tenant/project boundary.

Recommended fields:

```text
id
name
slug
description
owner_id
template_id nullable
status active|archived
created_at
updated_at
```

### WorkspaceMember

Controls workspace-level access.

Recommended roles:

```text
owner   full control, billing/settings/export/delete
admin   members, templates, documents, analysis, reports
editor  upload documents, run tasks, edit draft report content
reviewer review/approve/reject AI results
viewer  read approved content and reports
```

### WorkspaceTemplate

Defines reusable workspace structures.

Recommended fields:

```text
id
name
description
category
version
is_active
directory_schema jsonb
analysis_task_schema jsonb
report_schema jsonb
created_at
updated_at
```

Templates should be immutable by version once used. If a template changes, create a new version.

### WorkspaceDirectory

Represents logical document organization created from a template.

```text
id
workspace_id
parent_id nullable
name
path
sort_order
created_at
updated_at
```

### KnowledgeBase Changes

Add `workspace_id` to `knowledge_bases` and make it mandatory after migration.

Migration rule:

- For each v1.0 owner, create a default workspace.
- Attach that owner as workspace owner.
- Move each existing knowledge base into the owner's default workspace.
- Preserve existing `knowledge_base_members` during phase 1 for backward compatibility.
- Later, optionally migrate knowledge-base members into workspace members.

### Document and Chunk Changes

Add `workspace_id` to:

```text
documents
document_chunks
conversations
messages optional, if needed for query speed/audit
```

Even though documents and chunks can derive workspace through knowledge base, storing `workspace_id` directly prevents accidental cross-workspace joins and makes isolation tests simpler.

Every retrieval query must filter by both:

```text
workspace_id
knowledge_base_id or analysis scope
```

### AnalysisTask

Defines a specific AI job in a workspace.

```text
id
workspace_id
template_task_key
name
description
task_type extraction|classification|risk_review|summary|comparison|qa
status pending|running|completed|failed
input_scope jsonb
output_schema jsonb
created_by
created_at
updated_at
```

### AnalysisResult

Stores structured AI output with citations.

```text
id
workspace_id
analysis_task_id
status ai_generated|needs_review|approved|edited|rejected
result jsonb
citations jsonb
confidence numeric nullable
model
provider
token_usage jsonb
created_at
updated_at
```

Result JSON should be validated against the task output schema before persistence.

### ReviewDecision

Stores reviewer actions and preserves auditability.

```text
id
workspace_id
analysis_result_id
reviewer_id
decision approve|edit|reject|request_changes
comment nullable
original_result jsonb
edited_result jsonb nullable
created_at
```

Formal reports must read from approved/edited approved results only.

### Report and ReportSection

```text
reports:
  id
  workspace_id
  title
  status draft|ready|exported
  created_by
  created_at
  updated_at

report_sections:
  id
  report_id
  workspace_id
  title
  body_markdown
  source_result_ids uuid[] or jsonb
  sort_order
  status draft|approved
```

Report generation rule:

```text
Only approved or reviewer-edited approved AnalysisResult records may be included.
```

### ExportJob

```text
id
workspace_id
report_id
format markdown|docx|pdf
status pending|running|completed|failed
file_path nullable
error_message nullable
created_by
created_at
updated_at
```

### AuditLog

Track sensitive project actions:

```text
workspace.created
workspace.member_added
workspace.template_applied
document.uploaded
analysis_task.run
analysis_result.approved
analysis_result.rejected
report.exported
```

## Permission Rules

Workspace access should be checked before any nested resource access.

Minimum rules:

```text
owner/admin: manage workspace, members, templates, tasks, reports
editor: upload documents, run analysis, edit draft report sections
reviewer: review AI results, edit/reject/approve conclusions
viewer: read workspace, documents, approved reports, and conversations
```

Cross-workspace access must always return 404 or 403 consistently. Prefer 404 for resources the user should not know exist.

## AI Structured Output

v2.0 needs a new AI analysis layer beside the existing chat RAG layer.

Flow:

```text
AnalysisTask
-> retrieve workspace-scoped evidence
-> prompt LLM for strict JSON
-> parse JSON
-> validate schema
-> verify every conclusion has citations
-> save AnalysisResult as needs_review
```

Example output:

```json
{
  "findings": [
    {
      "claim": "Hotel reimbursement requires an itemized receipt and proof of payment.",
      "status": "supported",
      "confidence": 0.91,
      "citations": [
        {
          "document_id": "...",
          "chunk_id": "...",
          "page_number": 1,
          "quote": "Hotel reimbursement requires an itemized hotel receipt and proof of payment."
        }
      ]
    }
  ]
}
```

Implementation constraints:

- Reject malformed JSON.
- Reject uncited claims.
- Store raw model output for debugging if safe.
- Store normalized structured result for review and reporting.
- Keep deterministic fallback for tests.

## Migration Strategy

Create a v2 migration that is safe for existing v1.0 data.

High-level migration:

1. Create new tables: `workspaces`, `workspace_members`, `workspace_templates`, `workspace_directories`, `analysis_tasks`, `analysis_results`, `review_decisions`, `reports`, `report_sections`, `export_jobs`, `audit_logs`.
2. Add nullable `workspace_id` columns to `knowledge_bases`, `documents`, `document_chunks`, and `conversations`.
3. For every existing user that owns at least one knowledge base, create a default workspace.
4. Set `knowledge_bases.workspace_id` by owner default workspace.
5. Backfill `documents.workspace_id` from their knowledge base.
6. Backfill `document_chunks.workspace_id` from their document or knowledge base.
7. Backfill `conversations.workspace_id` from their knowledge base.
8. Make required `workspace_id` columns non-null after backfill.
9. Add indexes on all `workspace_id` columns.
10. Add foreign keys with cascade behavior where appropriate.

Downgrade should drop v2-only tables and workspace columns without deleting v1.0 core tables.

## Test Plan

Backend tests must cover:

- user can create multiple workspaces
- workspace template creates directories, tasks, and report sections
- workspace members enforce role permissions
- user cannot read another workspace
- user cannot upload documents into another workspace
- retrieval never returns chunks from another workspace
- conversations cannot cross workspace boundaries
- analysis results include citations
- malformed structured AI output is rejected
- reviewer can approve/edit/reject
- rejected results cannot enter formal reports
- exports only include approved content
- v1.0 data migration creates default workspaces and preserves knowledge bases, documents, chunks, conversations, and messages
- Docker fresh install runs all migrations
- Docker upgrade from v1.0 database runs successfully

Frontend tests/manual checks should cover:

- workspace creation with template selection
- dashboard status updates
- review queue decisions
- report preview and export actions
- cross-workspace navigation does not leak data

## Ten-Week Development Roadmap

The target schedule is about ten weeks. The plan keeps the stable v1.0 system on `main` and develops v2.0 on `feature/v2-workspaces`. Each week should end with passing backend checks, frontend build, and a small commit set that can be reviewed independently.

### Week 1 - Workspace Foundation

Goal: add workspace as the top-level project boundary without breaking v1.0 knowledge-base flows.

Daily plan:

```text
Day 1: Create Workspace, WorkspaceMember, and WorkspaceTemplate SQLAlchemy models and schemas.
Day 2: Add Alembic migration for workspace tables and indexes.
Day 3: Implement workspace CRUD services and API endpoints.
Day 4: Implement workspace member role checks and FastAPI dependencies.
Day 5: Seed built-in templates and expose template list/detail APIs.
Day 6: Add backend tests for workspace CRUD, ownership, member roles, and template listing.
Day 7: Update docs and run full backend checks.
```


Week 1 implementation status: completed on `feature/v2-workspaces`. The branch now includes workspace/template/member models, migrations, seeded templates, workspace CRUD APIs, member-management APIs, template list/detail APIs, and backend tests for ownership, roles, template routes, and access-control boundaries.

- Workspace owner membership is created automatically.
- Users only see workspaces they own or belong to.

## Schedule Risk and Buffer

The highest-risk items are:

- v1.0 data migration without loss
- cross-workspace isolation
- structured AI output validation
- report export quality for DOCX/PDF
- frontend workflow breadth

If the schedule slips, reduce scope in this order:

```text
1. Keep Markdown export complete; mark DOCX/PDF as beta if needed.
2. Keep built-in templates static; postpone user-created templates.
3. Keep analysis execution synchronous; postpone background workers.
4. Keep dashboard metrics simple; postpone charts.
```

Do not reduce scope on:

```text
workspace isolation
migration safety
review gating before formal reports
secret-free Docker deployment
```

## Initial Implementation Recommendation

Do not start by adding AI tasks or report exports. Start with Phase 1 and Phase 2 because every later feature depends on a trustworthy workspace boundary.

First implementation ticket:

```text
feat: add workspace models, CRUD APIs, member roles, and default templates
```

Second implementation ticket:

```text
feat: migrate knowledge bases, documents, chunks, and conversations into workspaces
```
